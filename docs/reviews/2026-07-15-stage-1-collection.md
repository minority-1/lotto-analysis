# 2026-07-15: 1단계 데이터 수집 코드 리뷰

## 결론

1단계의 정상 경로는 구현·자동 테스트·사용자 실행 확인을 마쳤다. 전체 원본 데이터도 현재 회차까지 수집되어 있다. 다만 비정상 공식 응답을 저장한 뒤 이어받기에서 완료로 오인할 수 있는 `P1` 결함이 한 건 확인되어, 2단계 구현 전에 보완해야 한다.

* `P0`: 없음
* `P1`: 1건
* `P2`: 2건
* `P3`: 2건

## 리뷰 범위

* 설정과 경로 관리
* 단일 회차 모델과 공식 수집기
* 재시도, 범위·전체·증분·누락 수집
* 원본 JSON과 충돌 파일 저장
* CLI, 진행률, 실행 이력과 회전 로그
* 단위·통합 테스트와 사용자 문서

## 발견 사항

### P1: 정규화 실패 원본이 유효한 저장 데이터로 판정될 수 있음

`DhlotteryDrawCollector.collect_draw()`는 공식 레코드를 저장한 다음 `LottoDraw`로 정규화한다. 원본에 `ltEpsd`만 있고 필수 번호·날짜·금액 필드가 빠졌다면 정규화는 실패하지만 파일은 남는다. `RawJsonStore.has_valid_draw()`는 JSON을 읽을 수 있고 `ltEpsd`가 일치하는지만 확인하므로, 이후 `collect-all` 이어받기와 `collect-missing`에서 해당 회차를 정상 완료로 오인한다.

리뷰 중 결함 입력을 임시 디렉터리에서 재현했으며 결과는 `ResponseFormatError`, `stored=True`, `has_valid_draw=True`였다. 현재 내려받은 정상 원본에서 발견된 데이터 오류는 아니고, 향후 응답 형식 변경 시 발생할 수 있는 복구 결함이다.

추천안: 공식 레코드를 먼저 정규화·검증하고 성공한 경우에만 정식 원본 경로에 저장한다. 실패 응답 자체를 보존해야 한다면 정식 원본이 아닌 별도의 오류 응답 경로에 저장한다. 필수 필드가 빠진 레코드가 이어받기 대상에서 제외되지 않는 회귀 테스트를 함께 추가한다.

### P2: 정수 변환이 실수를 조용히 절삭함

`_required_int()`가 모든 값을 `int(value)`로 변환하므로 `1.5`가 `1`로 처리된다. `LottoDraw`의 당첨자 수·금액 검증도 실제 `int` 타입인지 확인하지 않아 음수가 아닌 실수를 허용한다. 공식 응답이 정상인 동안에는 드러나지 않지만, 형식 변경이나 손상 데이터를 정상화해 버릴 수 있다.

추천안: `bool`과 실수를 거부하고 정수 또는 명확한 10진 정수 문자열만 허용한다. 모델에서도 회차, 번호, 인원, 금액의 실제 타입을 검증한다.

### P2: 초기화 단계 실패는 실행 이력에 기록되지 않음

설정 읽기, 로깅 구성, 저장소·수집기·서비스 생성이 CLI의 실행 이력 `try` 범위보다 앞에 있다. 따라서 잘못된 환경변수나 로그 파일 생성 실패는 `data/collection_history/`에 실패 이력을 남기지 않는다. `모든 CLI 수집 실행 결과`를 기록한다는 문서 표현과 실제 동작에 차이가 있다.

추천안: 명령 파싱 뒤 초기화와 실행을 하나의 상위 예외 경계에서 관리한다. 다만 이력 디렉터리 자체를 만들 수 없는 오류는 저장할 수 없으므로, 이 제한은 stderr와 문서에 명시한다.

### P3: 동시 실행 시 임시 파일 이름이 충돌할 수 있음

원본과 실행 이력의 원자적 쓰기가 항상 같은 `.tmp` 파일명을 사용하며 실행 이력의 고유 경로 선택도 확인 후 생성 방식이다. CLI 두 개를 동시에 실행하면 임시 파일이나 이력 파일을 서로 덮을 가능성이 있다.

추천안: 현재는 단일 프로세스 운영을 전제로 문서화한다. 병렬 수집이 필요해질 때 프로세스 잠금 또는 `tempfile` 기반 고유 임시 파일과 배타적 생성을 도입한다.

### P3: Windows 호환성이 자동 검증되지 않음

경로 처리는 `pathlib`을 사용하고 Windows 실행 방법도 제공하지만 현재 자동 테스트는 macOS 환경에서만 확인했다.

추천안: GitHub Actions를 도입할 때 Python 3.9 이상과 macOS/Windows 조합을 최소 매트릭스로 추가한다.

## 확인된 정상 설계

* 수집기, 저장소, 서비스, CLI 책임이 분리되어 핵심 로직이 UI에 종속되지 않는다.
* 재시도 대상이 일시적 네트워크 오류로 제한되어 영구 오류를 불필요하게 반복하지 않는다.
* 기존 원본 변경 시 자동 덮어쓰지 않고 충돌 원본과 변경 필드를 보존한다.
* 증분 수집과 누락 보완을 별도 명령으로 둔 동작이 문서·테스트와 일치한다.
* 설정값과 경로가 환경변수로 변경 가능하고 `pathlib`을 사용한다.
* 외부 네트워크 테스트가 기본 테스트와 분리되어 있다.

## 테스트 및 검증

* 자동 테스트: macOS, Python 3.9.6에서 `64 passed, 2 skipped` (`pytest-8.4.2`).
* 외부 테스트: 기본 실행에서는 제외하며, 공식 사이트 확인이 필요할 때 선택 실행한다.
* 사용자 검증: 진행률, 실행 이력과 파일 로그의 정상 작동을 사용자가 확인했다.
* 데이터 상태: 1회부터 당시 최신 회차까지 전체 원본 수집을 사용자가 완료했다.

## 종료 판단

1단계 기능 개발과 운영 확인은 종료한다. `P1`은 2단계 데이터 정제·SQLite 작업에 들어가기 전 별도의 작은 보완 작업으로 처리한다. `P2`는 같은 보완 작업에 포함하는 것을 권장하고, `P3`는 병렬 실행 또는 CI 도입 시점까지 보류할 수 있다.
