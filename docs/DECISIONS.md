# Project Decisions

이 문서는 프로젝트를 진행하면서 확정한 설계 및 기술 결정을 기록한다.

## 2026-07-14: 1단계 기본 골격

* Python 패키지는 `src/lotto_analysis` 레이아웃을 사용한다.
* 프로젝트 메타데이터와 pytest 설정은 `pyproject.toml`에서 관리한다.
* 설정 및 경로 관리는 표준 라이브러리와 `pathlib`을 사용하며, 환경변수로 경로를 변경할 수 있게 한다.
* 정규화된 단일 회차 모델은 불변 `dataclass`로 표현한다.
* 단일 회차 수집기는 추상 기반 클래스 인터페이스로 정의하고, 실제 외부 데이터 출처 구현은 후속 작업에서 결정한다.
* logging은 표준 라이브러리를 사용하고 콘솔 출력을 기본으로 하며 파일 출력은 선택적으로 설정한다.

## 2026-07-14: 공식 단일 회차 데이터 출처

* 단일 회차 데이터는 동행복권 공식 결과 화면이 사용하는 JSON 엔드포인트 `/lt645/selectPstLt645InfoNew.do`에서 조회한다.
* 요청에는 `srchDir=center`와 `srchLtEpsd=<회차>`를 사용하고, 응답 목록에서 요청 회차를 다시 확인하여 정확히 한 건만 반환한다.
* 해당 엔드포인트의 공개 API 문서는 확인되지 않았으므로 응답 구조 변경 가능성을 전제로 명시적인 형식 오류를 제공한다.
* HTTP 클라이언트는 명세에 포함된 `requests`를 사용하며 URL, timeout, User-Agent는 설정으로 관리한다.
* macOS 기본 Python 3.9의 LibreSSL 호환성을 위해 `urllib3`는 1.26 계열로 제한한다.
* 네트워크 오류, 회차 없음, 응답 형식 오류를 서로 다른 수집 예외로 구분한다.

## 2026-07-14: 범위 수집과 원본 보존

* 최신 회차는 공식 결과 페이지의 `opt_val` 입력값에서 확인하며 임의의 미래 회차를 조회해 추측하지 않는다.
* 범위 수집은 회차별 독립 요청으로 실행하고 기본 요청 간격은 0.5초로 둔다.
* 원본 JSON은 요청 회차의 공식 레코드 한 건을 `data/raw/draw_NNNN.json`으로 저장한다.
* 한 회차의 알려진 수집·파일 오류는 실행 요약에 기록하고 다음 회차 수집을 계속한다.

## 2026-07-15: 복구 가능한 전체 수집

* timeout, 연결 오류, HTTP 429와 5xx만 자동 재시도하며 기본 최대 재시도 횟수는 3회다.
* 재시도 대기는 기본 0.5초에서 시작해 시도마다 두 배로 증가한다.
* 전체·범위 수집은 기존의 유효한 원본 파일을 기본적으로 건너뛰어 중단 후 이어받을 수 있게 한다.
* 증분 수집은 저장된 최대 회차 다음부터 최신 회차까지 수집한다.
* 누락 보완은 1회부터 최신 회차 사이의 없거나 유효하지 않은 파일만 수집한다.
* 기존 원본과 공식 레코드가 다르면 기존 파일은 유지하고 신규 레코드를 `data/raw/conflicts/`에 내용 해시 기반 파일명으로 저장한다.
* 충돌 결과에는 값이 달라진 필드명을 포함하며 자동으로 어느 쪽을 정본으로 선택하지 않는다.

## 2026-07-15: 수집 실행 이력

* 모든 CLI 수집 실행 결과는 `data/collection_history/`에 실행별 JSON 파일로 저장한다.
* 실행 이력에는 명령, 시작·종료 시각, 소요 시간, 대상 범위, 성공·건너뜀·실패 회차와 실패 사유를 기록한다.
* 최신 회차 조회처럼 회차 수집 전에 실행 자체가 실패한 경우에도 오류 이력을 저장한다.
* 실행 이력 파일은 기존 파일을 덮어쓰지 않으며 수집 원본과 마찬가지로 Git에 포함하지 않는다.

## 2026-07-15: 수집 진행률과 파일 로그

* 수집 진행 상태는 Service의 선택적 콜백으로 제공하고 CLI가 사용자 출력 형식을 담당한다.
* 긴 수집 실행은 약 1% 단위로 진행률을 표시하며 실패 상태는 즉시 표시한다.
* 애플리케이션 로그는 콘솔과 `logs/lotto-analysis.log`에 함께 기록한다.
* 파일 로그는 기본 5 MiB 크기와 백업 3개를 사용하는 크기 기반 회전 정책을 적용한다.
* 로그 경로, 최대 크기와 백업 개수는 환경변수로 변경할 수 있다.

## 2026-07-15: 사용자 확인 작업 관리

* 사용자가 직접 실행·확인·결정해야 하는 작업은 `docs/USER_CHECKLIST.md`에 누적 기록한다.
* 체크리스트에는 현재 상태, 실행 명령, 예상 확인 결과와 반복 운영 작업을 함께 기록한다.
* 개발 단계가 바뀌거나 새로운 사용자 판단이 필요해질 때 체크리스트도 함께 갱신한다.

## 2026-07-15: 단계별 코드 리뷰 기록

* 주요 개발 단계가 끝날 때 `docs/reviews/`에 범위, 발견 사항, 테스트와 종료 판단을 기록한다.
* 발견 사항은 `P0`부터 `P3`까지 우선순위를 부여하고, 해결하지 않은 항목은 다음 단계 전 체크리스트에 연결한다.
* 코드 리뷰와 코드 수정은 분리하며, 리뷰 요청만 받은 경우에는 발견 사항을 임의로 수정하지 않는다.

## 2026-07-15: 1단계 리뷰 보완

* 공식 원본의 정식 저장은 필수 필드 정규화와 도메인 모델 검증이 성공한 뒤 수행한다.
* 공식 정수 필드는 실제 정수 또는 명확한 10진 정수 문자열만 허용하며 실수의 암묵적 절삭을 금지한다.
* 정규화 모델은 날짜, timezone 포함 시각과 숫자 필드 타입을 실행 시점에 검증한다.
* CLI는 설정 해석 이후의 초기화·수집 오류를 실패 이력에 기록한다. 설정 자체를 해석할 수 없거나 이력 저장소에 접근할 수 없으면 원래 오류와 이력 저장 불가 사유를 터미널에 표시한다.
* 동시 프로세스 파일 잠금과 Windows CI는 현재 단일 사용자·단일 프로세스 1단계 범위 밖이므로 후속 작업으로 유지한다.

## 2026-07-15: SQLite 제외와 정제 CSV

* 초기 SQLite 구현은 제외하고 원본 JSON에서 정제 CSV를 생성한 뒤 기본 분석을 진행한다.
* 데이터베이스가 필요한 시점에는 SQLite 중간 구현 대신 PostgreSQL과 Repository 계층을 도입한다.
* 정제 작업은 매번 전체 원본을 검증하여 `data/processed/lotto_draws.csv`를 재생성한다.
* 잘못된 원본이 있어도 정상 회차는 CSV로 생성하고 오류·중복·누락은 `validation_report.json`에 분리한다. 오류나 누락이 있으면 CLI는 실패 종료 코드를 반환한다.
* 기존 공식 원본에는 수집 시각이 없으므로 CSV의 `collected_at`을 빈 값으로 유지하며 파일 수정 시각으로 추정하지 않는다.

## 2026-07-17: 1차 기본 통계 분석

* 분석 엔진은 저장 방식에 종속되지 않으며 `DrawRepository` 인터페이스로 정제 데이터를 받는다.
* 최초 구현은 `CsvDrawRepository`를 사용하고 PostgreSQL 도입 시 Repository 구현을 교체한다.
* 출현 비율은 해당 번호의 일반번호 출현 횟수를 분석 대상 회차 수로 나눈 값으로 정의한다.
* 미출현 기간은 선택한 분석 범위에서 해당 번호의 마지막 출현 이후 지난 회차 수로 정의한다.
* 저번호와 고번호는 `1~22 / 23~45`, 번호 구간은 `1~10 / 11~20 / 21~30 / 31~40 / 41~45`로 구분한다.
* 연속번호 개수는 정렬된 당첨번호에서 차이가 1인 인접 번호쌍의 개수다.
* 분석 결과는 실행 시 계산하고 아직 별도 파일이나 데이터베이스에 저장하지 않는다.
* 모든 분석 결과는 과거 데이터의 기술통계이며 예측이나 당첨 가능성으로 표현하지 않는다.

## 2026-07-17: PostgreSQL 전 분석 범위 완료

* 기본 분석은 합계의 최소·최대·평균·중앙값·표준편차, 홀짝·저고·구간·끝수 분포와 연속번호 비율을 요약한다.
* 기간 비교 기본값은 최근 N회와 직전 N회이며 선택적으로 전체 기간과 최근 N회를 비교한다.
* 기간 비교는 출현 횟수와 함께 회차당 출현률 차이를 사용한다.
* 번호 순위는 `1 + 자신보다 출현률이 높은 번호 개수`로 정의하여 동률 번호에 같은 순위를 부여한다.
* 출현 간격은 실제 회차 번호 차이를 사용하고 평균·중앙·최소·최대·최근 간격과 모집단 표준편차를 계산한다.
* 쉽게 재계산 가능한 분석 결과는 데이터베이스에 저장하지 않으며 사용자가 `--export`를 지정할 때만 `data/analysis/`에 JSON으로 저장한다.
* 번호쌍·3개 조합·행렬·조건 생성·백테스트는 PostgreSQL 도입 이후 단계로 둔다.

## 2026-07-17: Docker 기반 PostgreSQL 개발 환경

* 로컬 PostgreSQL은 Docker Desktop과 Docker Compose로 실행한다.
* PostgreSQL 17 공식 Alpine 이미지를 사용하고 최신 17.x 수정 버전은 이미지 갱신 시 반영한다.
* DB 포트는 외부 네트워크에 공개하지 않고 `127.0.0.1`에만 연결한다.
* DB 파일은 Compose 명명 볼륨에 보존하여 컨테이너를 다시 만들어도 유지한다.
* DB 이름, 사용자, 비밀번호와 호스트 포트는 `.env`에서 읽고 실제 `.env`는 Git에 포함하지 않는다.
* Compose 설정과 컨테이너 검증을 먼저 마친 뒤 SQLAlchemy, psycopg, Alembic과 Repository를 별도 작업으로 도입한다.

## 2026-07-17: PostgreSQL 스키마와 Repository

* PostgreSQL 연결과 ORM은 SQLAlchemy 2.x, 드라이버는 psycopg 3.x를 사용한다.
* 스키마 버전은 Alembic 마이그레이션으로 관리하며 애플리케이션 시작 시 암묵적으로 테이블을 생성하지 않는다.
* 정규화된 회차 데이터는 `lotto_draws` 한 테이블에 저장하고 회차 번호를 유일 키로 유지한다.
* 번호 범위·정렬·중복, 보너스 번호 중복과 음수 금액은 데이터베이스 제약조건으로도 방어한다.
* `PostgresDrawRepository`는 기존 `DrawRepository`를 구현하여 분석 엔진이 저장 방식에 종속되지 않게 한다.
* 최초 이관은 검증된 CSV 전체를 회차 번호 기준 upsert하고, 재실행해도 중복 행이 생기지 않게 한다.
* 이관 후 CSV와 PostgreSQL의 정규화 데이터 및 기본 분석 결과를 함께 비교한다.
* DB GUI는 무료 오픈소스와 다중 DB 지원을 우선해 DBeaver Community를 권장하되 필수 개발 의존성으로 두지 않는다.

## 2026-07-17: 관계 분석 1차 범위

* 관계 분석의 첫 범위는 일반 당첨번호 6개 사이의 번호쌍, 3개 조합과 특정 번호의 동반 출현으로 제한한다.
* 분석 Service는 `DrawRepository`에만 의존하고 CLI에서는 PostgreSQL Repository를 사용한다.
* 출현하지 않은 모든 가능한 조합을 0건으로 생성하지 않고 실제 등장한 조합만 반환한다.
* 조합 출현률은 해당 조합 출현 회차 수를 분석 대상 회차 수로 나눈다.
* 특정 번호의 동반 출현률은 상대 번호 동시 출현 횟수를 기준 번호 출현 횟수로 나눈 조건부 비율이다.
* 결과는 횟수 내림차순, 동률이면 번호 오름차순으로 정렬한다.
* 재계산 가능한 관계 분석 결과는 DB에 저장하지 않고 요청 시에만 JSON으로 내보낸다.
* 동시 출현 빈도와 조건부 비율은 과거 관계를 설명할 뿐 미래 당첨 확률이나 인과관계를 의미하지 않는다.

## 2026-07-17: 관계 분석 2차 범위

* 번호 간 거리는 한 회차의 일반번호 6개에서 만들어지는 15개 번호쌍의 절대 차이로 정의한다.
* 거리 비율의 분모는 분석 회차 수의 15배인 전체 번호쌍 관측 수다.
* `±1 관계`는 거리가 1인 번호쌍으로 정의하고 총 번호쌍 수와 하나 이상 포함한 회차 수를 함께 제공한다.
* 같은 끝수 관계는 두 번호를 10으로 나눈 나머지가 같은 번호쌍으로 정의한다.
* 연속번호 그룹은 한 회차 안에서 길이가 2 이상인 최대 연속 구간으로 집계한다.
* 이전 회차 중복은 분석 범위 안에 정확한 `현재 회차 - 1·2·3` 회차가 있을 때만 비교한다.
* 이전 회차 중복은 0~6개 분포와 비교 대상당 평균 중복 개수를 반환한다.

## 2026-07-17: 보너스 번호 후속 출현 분석

* 한 회차의 보너스 번호가 정확히 1·2·3회 뒤 회차의 일반번호 6개에 포함됐는지 각각 집계한다.
* 분석 범위 안에 정확한 후속 회차 번호가 있을 때만 해당 기준 회차를 분모에 포함한다.
* 출현 비율은 후속 일반번호 출현 횟수를 비교 가능한 기준 회차 수로 나눈다.
* 보너스 번호와 같은 회차의 일반번호 관계나 3회를 초과한 누적 출현은 이번 범위에 포함하지 않는다.
* 결과는 과거 역할 전환 빈도이며 이후 번호 출현을 예측하는 지표로 사용하지 않는다.

## 2026-07-17: 7×7 번호 행렬 기본 분석

* 번호 1~45는 1부터 행 우선 순서로 7×7 고정 좌표에 배치한다.
* 마지막 행의 46~49 위치는 번호가 없는 빈 셀로 유지하고 집계에서 제외한다.
* 셀은 일반 당첨번호 출현 횟수와 분석 회차당 출현 비율을 제공한다.
* 행·열 통계는 번호 출현 횟수의 합계와 회차당 사용한 서로 다른 행·열 개수의 평균을 제공한다.
* 전체와 최근 N회는 같은 계산을 사용하며 CLI에서는 PostgreSQL Repository를 사용한다.
* 대각선과 두 기간 차이 행렬은 기본 행렬 검증 후 다음 범위에서 구현한다.
* 행렬 분포는 과거 데이터의 공간적 표현이며 미래 당첨 가능성을 의미하지 않는다.

## 2026-07-17: 행렬 대각선과 기간 비교

* 주대각선은 유효한 셀 `1, 9, 17, 25, 33, 41`, 역대각선은 `7, 13, 19, 25, 31, 37, 43`으로 정의한다.
* 대각선 통계는 번호별 합산 출현 수와 하나 이상의 대각선 번호를 포함한 회차 수·비율을 제공한다.
* 행렬 기간 비교의 기본 방식은 최근 N회와 바로 앞의 직전 N회다.
* 비교 결과는 49개 고정 셀마다 두 기간의 출현 횟수·출현률과 `최근 출현률 - 직전 출현률`을 기록한다.
* 두 기간을 온전히 구성할 수 없으면 비교하지 않고 오류를 반환한다.
* 출현률 차이는 과거 두 기간의 기술적 차이이며 추세나 미래 예측으로 해석하지 않는다.

## 2026-07-17: 조합 수학 특성 분석

* AC 값은 한 조합의 15개 번호쌍에서 얻은 서로 다른 양의 차이 개수에서 5를 뺀 값으로 정의한다.
* 번호 간격은 정렬된 일반번호 6개 사이의 인접한 5개 차이로 정의한다.
* 소수는 1~45의 표준 소수 14개이며 숫자 1은 소수와 합성수 어디에도 포함하지 않는다.
* 제곱수는 `1, 4, 9, 16, 25, 36`으로 정의하고 개수와 포함 여부를 함께 기록한다.
* 합계 구간은 `21-100 / 101-120 / 121-140 / 141-160 / 161-180 / 181-255`를 사용한다. 중앙부는 20 단위, 양쪽 희소 범위는 하나로 묶는다.
* 끝수 합계는 각 번호를 10으로 나눈 나머지 6개의 합으로 정의한다.
* 회차별 상세 특성과 전체 분포를 반환하고, 요청 시 JSON으로만 저장한다.
* 수학 특성은 이후 조합 필터에서 재사용할 수 있지만 그 자체가 당첨 확률을 높인다고 해석하지 않는다.

## 2026-07-17: 과거 당첨 조합 유사도

* 각 회차는 선택된 분석 범위 안에서 회차 번호가 작은 이전 회차들과만 비교한다.
* 전체 분포는 같은 회차쌍을 중복 계산하지 않는 순서 없는 조합으로 집계한다.
* 유사도는 공통 일반번호 개수와 Jaccard 지수 `교집합 크기 ÷ 합집합 크기`를 함께 사용한다.
* 회차별로 최대 중복 번호 수, 최대값을 공유하는 모든 이전 회차와 정확히 3·4·5·6개가 겹친 이전 회차 수를 기록한다.
* `recent N`은 최근 N회 내부 유사도만 계산하며 선택 범위 밖 데이터는 암묵적으로 포함하지 않는다.
* 전체 회차쌍 목록은 메모리에 보존하지 않고 집계값과 회차별 요약만 반환한다.
* 유사도는 조합 생성 시 과거 동일·유사 조합 제외에 활용할 수 있지만 당첨 가능성 점수로 해석하지 않는다.

## 2026-07-17: 최근 범위 초과 요청 처리

* 단일 기간 분석의 `recent N`이 실제 보유 회차 수보다 크면 전체 데이터로 대체하지 않고 오류를 반환한다.
* 오류에는 요청 건수와 실제 사용 가능한 회차 수를 함께 표시한다.
* 기본 분석, 출현 간격, 관계, 행렬, 조합 수학 특성과 유사도에 같은 기준을 적용한다.

## 2026-07-17: 조건 기반 번호 생성 1차 범위

* 생성 전략과 조건 필터는 별도 인터페이스와 모듈로 분리한다.
* 첫 전략은 Python `random.Random`을 사용하는 균등 무작위 비복원 추출이며 선택적 seed로 재현한다.
* 첫 조건 범위는 포함·제외 번호, 홀수, 낮은 번호, 합계, 소수, AC, 연속번호쌍과 과거·결과 조합 중복 제한이다.
* 낮은 번호는 기존 분석과 같은 1~22를 사용하며 짝수·높은 번호 개수는 홀수·낮은 번호 개수에서 파생한다.
* 기본적으로 과거 완전 동일 조합과 과거·생성 결과에 5개 이상 겹치는 조합을 제외한다.
* 기본 최대 시도는 10000회이며 미완료 시 생성된 결과는 유지하고 주요 거절 조건을 설명한다.
* 조건을 통과한 조합 사이의 근거 있는 우열 공식이 아직 없으므로 1차 범위에서는 점수를 만들거나 정렬하지 않는다.
* 생성 결과에는 홀짝·고저·합계·소수·AC·연속번호와 과거 최대 중복을 함께 제공한다.
* 생성 조합은 사용자 조건을 만족하는 후보일 뿐 추천 점수나 당첨 확률을 의미하지 않는다.

## 2026-07-17: 빈도 가중 번호 생성 전략

* 빈도 가중 전략은 전체 또는 최근 N회의 일반번호 출현 횟수를 사용한다.
* 기본 가중치는 `출현 횟수 + 1` additive smoothing으로 0 가중치를 방지한다.
* 각 가중치는 45개 번호 평균 가중치의 0.5배~1.5배 범위로 제한한다.
* 후보 6개는 제한된 가중치를 사용해 비복원 추출하며 필수·제외 번호와 기존 필터를 그대로 적용한다.
* seed, 가중치 원본 회차 수, smoothing과 cap 공식을 결과에 기록한다.
* 내보내기 파일명은 전략과 전체·최근 가중 범위를 포함해 같은 seed의 다른 전략 결과를 덮어쓰지 않는다.
* 빈도 가중치는 사용자 선택 전략이며 미래 당첨 확률 추정치로 표현하지 않는다.
